# 📘 Chapter 3: SHA-1 Hashing

## Why SHA-1?

BitTorrent uses SHA-1 (Secure Hash Algorithm 1) for two purposes:
1. Computing the **info hash** (torrent's unique ID)
2. Verifying **each downloaded piece** for integrity

SHA-1 produces a 160-bit (20-byte) hash from any input. The same input
always produces the same output, and it's computationally infeasible to
find two inputs that produce the same hash (collision resistance).

> Note: SHA-1 is considered cryptographically weak today (collisions have
> been found), but BitTorrent doesn't need collision resistance for security —
> just for integrity verification of known data.

---

## How SHA-1 Works (Conceptually)

SHA-1 processes data in **64-byte (512-bit) chunks**:

```
Input message
     ↓
[Padding to multiple of 512 bits]
     ↓
Split into 512-bit blocks
     ↓
For each block:
  Expand 16 words → 80 words
  Run 80 rounds of mixing with 5 state variables (a,b,c,d,e)
  Add results back to running state
     ↓
Final state = 5 × 32-bit words = 160-bit (20-byte) hash
```

### The 5 State Variables

SHA-1 maintains 5 32-bit integers as its running state, initialized to
magic constants from the spec:

```c
uint32_t h0 = 0x67452301;
uint32_t h1 = 0xEFCDAB89;
uint32_t h2 = 0x98BADCFE;
uint32_t h3 = 0x10325476;
uint32_t h4 = 0xC3D2E1F0;
```

These are the fractional parts of square roots of primes — a "nothing up
my sleeve" technique to prove the constants aren't backdoored.

---

## Padding Rules

Before processing, the message is padded to a multiple of 512 bits:

1. Append a single `1` bit (byte `0x80`)
2. Append `0` bits until message length ≡ 448 (mod 512)
3. Append the original message length as a 64-bit big-endian integer

```
[original message] [0x80] [0x00...] [length as 8 bytes]
                                    ←  64 bytes total chunk  →
```

---

## The 80 Rounds

Each 512-bit block is expanded from 16 words (W[0..15]) to 80 words:

```c
for (int i = 16; i < 80; i++) {
    W[i] = left_rotate(W[i-3] ^ W[i-8] ^ W[i-14] ^ W[i-16], 1);
}
```

Then 80 rounds of mixing, in 4 groups of 20:

| Rounds | Function f(b,c,d)           | Constant K    |
|--------|-----------------------------|---------------|
| 0–19   | `(b & c) \| (~b & d)`      | `0x5A827999`  |
| 20–39  | `b ^ c ^ d`                 | `0x6ED9EBA1`  |
| 40–59  | `(b&c) \| (b&d) \| (c&d)` | `0x8F1BBCDC`  |
| 60–79  | `b ^ c ^ d`                 | `0xCA62C1D6`  |

Each round:
```c
uint32_t temp = left_rotate(a, 5) + f(b,c,d) + e + K + W[i];
e = d; d = c; c = left_rotate(b, 30); b = a; a = temp;
```

---

## Big-Endian Byte Order

SHA-1 uses **big-endian** byte order. On x86 (little-endian) machines,
you must byte-swap 32-bit words when reading input and writing output.

```c
// Big-endian read of 4 bytes
uint32_t w = ((uint32_t)buf[0] << 24) |
             ((uint32_t)buf[1] << 16) |
             ((uint32_t)buf[2] <<  8) |
             ((uint32_t)buf[3]);
```

## Using SHA-1 in This Client

```c
uint8_t hash[20];

// Hash a complete buffer at once
sha1(data, data_len, hash);

// Incremental hashing (for streaming)
SHA1_CTX ctx;
sha1_init(&ctx);
sha1_update(&ctx, chunk1, len1);
sha1_update(&ctx, chunk2, len2);
sha1_final(&ctx, hash);
```

---

## Verifying a Piece

```c
// After downloading piece i:
uint8_t computed[20];
sha1(piece_data, piece_length, computed);

uint8_t *expected = torrent->pieces_hash + (i * 20);

if (memcmp(computed, expected, 20) == 0) {
    printf("Piece %d verified ✓\n", i);
} else {
    printf("Piece %d CORRUPTED — re-downloading\n", i);
}
```
